StringTableResource

When dealing with string tables, you'll need to use the StringTableResource class from @s4tk/models.

Creating STBL

To create a new, empty string table, you can use new StringTableResource().

Adding Strings

The addAndHash(string) method will add the given string to the table, and use its 32-bit hash as its key. While convenient, it may not be the best idea to use this function in production - you want to make sure that your string keys are unique.

The addAndHash(string, toHash) method will add the given string to the table, and use the 32-bit hash of the toHash argument as its key. This is a bit "safer" than hashing the string itself, because you can append your creator name to the front, increasing the liklihood that your key will be unique.

Finally, the add(key, string) method will add the given string to the table with the given key. You can put in a fixed value for the key, or you can use the fnv32() function from @s4tk/hashing to generate a hash manually.

Getting Strings

You can get a string entry by its key with the getByKey(key) method, or by its value with the getByValue(value) method. Note that the value must be an exact match, case sensitive, in order to return the entry. The entry that is returned is an object with key and value properties.

Additionally, you can use the entries property, which is a list of all of the entries in the string table. You can use this list to get entries by their index, or to loop/iterate over all of the entries. Note that adding/removing entries to/from this list will NOT add/remove them from the string table - you MUST use the add/delete methods.

Editing Strings

Once you get a reference to an entry, editing it is as easy as setting the key and value properties.

Deleting Strings

To delete an entry with a known key, you can use the deleteByKey(key) method. If you don't know the key, though, you can fetch the entry you want to delete (by its value or index), and then delete it by its unique ID with delete(id).

If you're confused what the difference between a key and an id is, it's quite simple. Keys are what the game uses, they identify which strings are which. IDs are what S4TK uses, they identify which entries are which. IDs are not a part of the string table, they are just used by S4TK to ensure uniqueness, since keys are NOT guaranteed to be unique (they should be, but it is technically valid for them to overlap).
