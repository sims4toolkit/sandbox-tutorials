Model Bases

While not something you'd normally concern yourself with, it may be worth reading about the base classes that various models descend from, since they define common behavior.

Don't feel like you have to remember or even understand the code presented for this tutorial. The sole purpose here is to demonstrate how similar two completely different models can be, thanks to the base classes they inherit from.

Writable Models

The WritableModel base class defines behavior that is shared between every model that can be written to disk.

One of the most important methods to know about is the static from(buffer: Buffer) method that every model implements. It accepts a buffer of data to load into the model, such as a binary STBL or plain text XML.

Another important method to know about is getBuffer(), which returns an uncompressed buffer of the model's data - this is the inverse of the from() method.

Check the docs for more information about the optional arguments that from() and getBuffer() accept. For most scripts, you do not need to worry about these options.

Mapped Models

The MappedModel base class inherits from WritableModel, and is used for models that map keys to values. At the time of writing, the two models that do so are Package and StringTableResource.

The vast majority of methods you'll use to interact with packages and string tables come from this base class, so I highly recommend that you read through the docs.

A key/value pair in a mapped model is called an entry. The most important thing to note about entries are their key, value, and id properties.

Mapped models are lists of key/value pairs, not actual mappings - therefore, keys are allowed to repeat. S4TK uses an internal id property that acts as a unique, constant identifier, but this ID does not exist in the actual file binaries.

You generally do not have to worry about it, just know that it exists and is the preferred way to get and delete entries.
